# 05 - Consistency And Resilience Guide

## 1. CDP 是否有分布式事务场景

结论：

1. **有跨服务一致性问题**。
2. **通常不使用强一致分布式事务（2PC/全局锁）**。
3. 主流方案是最终一致：幂等 + 重试 + 补偿 + 事件驱动。

## 2. 为什么 CDP 常见链路不做强事务

1. 数据面吞吐高，强事务代价太大。
2. 流式链路天然异步，跨系统延迟不可控。
3. 标签、画像、分群大多允许短时最终一致。

## 3. 哪些地方可能出现“可选强事务”

1. 低 QPS 的控制面配置变更。
2. 同库内状态机更新。
3. 极少数强约束运营动作（通常在 MA/权益系统，不是 CDP 主链路）。

## 4. CDP 有没有限流/熔断/降级场景

结论：**有，而且必须有**。

## 5. 限流场景

1. OpenAPI 查询接口防止单租户打爆。
2. 演示入口批量触发接口防止误操作风暴。
3. 下游慢时限制并发，保护线程池和连接池。

推荐策略：

1. 租户维度 QPS。
2. 接口维度令牌桶。
3. 单 IP 短时突发保护。

## 6. 熔断场景

1. `callchain-demo -> connector-control` 下游超时。
2. `connector-control -> flink-job` 下游故障。
3. `tag-task -> flink-job` 状态查询大量失败。

推荐策略：

1. 按接口设置失败率阈值触发熔断。
2. 熔断打开后快速失败，避免级联阻塞。
3. 熔断半开后探测恢复。

## 7. 降级场景

1. MySQL 不可用时，ODS 查询降级到内存快照。
2. 下游不可用时只返回“任务已受理”，异步补偿。
3. 可观测组件不可用时，业务链路继续运行。

## 8. 一致性落地建议

1. 为每个任务和事件生成全局业务 ID。
2. 消费端先幂等校验再执行业务。
3. 失败事件进入重试队列或死信队列。
4. 定时对账并做补偿重放。

## 9. 与当前演示实现的对应

1. `cdp-flink-job-service` 已支持状态机和手工状态推进。
2. ODS 已支持 MySQL 优先、内存兜底。
3. 后续可在此基础上补充 Sentinel/Resilience4j 策略演示。
